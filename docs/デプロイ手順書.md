# デプロイ手順書

## 1. 前提条件

| 項目 | 要件 |
|------|------|
| Terraform | >= 1.7 |
| AWS CLI | 設定済み（適切な権限を持つプロファイル） |
| Node.js | 20以上 |
| PowerShell | 5.1以上（Windowsの場合） |

### 必要なAWS権限

- Lambda: `UpdateFunctionCode`, `GetFunction`
- S3: `PutObject`, `DeleteObject`, `ListBucket`, `GetBucketLocation`
- CloudFront: `CreateInvalidation`
- DynamoDB: `PutItem`, `GetItem`, `DeleteItem`, `Scan`（Lambda実行ロールに付与済み）

---

## 2. 初回インフラ構築（Terraform）

初回のみ実行する。2回目以降はスキップ。

```powershell
cd infra/

# 初期化（S3バックエンド・プロバイダーダウンロード）
terraform init

# 変更内容の確認
terraform plan

# 適用（CloudFront作成に5〜10分かかる）
terraform apply
```

**出力値の確認**

```powershell
terraform output
```

主要な出力値：

| 出力名 | 用途 |
|-------|------|
| `cloudfront_domain` | フロントエンドURL（例: `d2b2m401smddic.cloudfront.net`） |
| `cloudfront_distribution_id` | キャッシュ無効化に使用 |
| `api_gateway_endpoint` | ローカル開発時の `.env.local` に設定 |
| `frontend_bucket_name` | フロントエンドデプロイ先バケット名 |

## 3. Lambda のデプロイ（バックエンド更新）

`backend/lambda/handler.rb` を変更した場合に実行する。

### Step 1: zipファイルの作成

```powershell
# PowerShell（Windowsの場合）
Compress-Archive -Path backend/lambda/* -DestinationPath infra/modules/lambda/lambda.zip -Force
```

### Step 2: Lambda コードの更新

```powershell
aws lambda update-function-code `
  --function-name recipe-app-api `
  --zip-file fileb://infra/modules/lambda/lambda.zip
```

### Step 3: 動作確認

```powershell
# ヘルスチェック
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
Invoke-RestMethod -Uri "https://d2b2m401smddic.cloudfront.net/health" -Method GET
```

期待レスポンス: `{"status":"ok"}`

---

## 4. フロントエンドのデプロイ（Next.js更新）

`frontend/` 以下を変更した場合に実行する。

### Step 1: ビルド

```powershell
cd frontend/
npm run build
```

`out/` ディレクトリに静的ファイルが生成される。

### Step 2: S3 にアップロード

```powershell
# バケット名は terraform output frontend_bucket_name で確認
aws s3 sync out s3://<frontend-bucket-name> --delete
```

> `--delete` オプションにより、S3上の不要な古いファイルも削除される。

### Step 3: CloudFront キャッシュの無効化

```powershell
# Distribution IDは terraform output cloudfront_distribution_id で確認
aws cloudfront create-invalidation `
  --distribution-id <distribution-id> `
  --paths "/*"
```

無効化には1〜2分かかる。完了後にブラウザでアクセスして動作確認する。

### Step 4: 動作確認

ブラウザで `https://d2b2m401smddic.cloudfront.net` にアクセス。

## 6. ローカル開発環境のセットアップ

### フロントエンドのローカル起動

```powershell
cd frontend/

# 依存パッケージのインストール（初回のみ）
npm install

# 環境変数の設定（API GatewayのエンドポイントURL）
# terraform output api_gateway_endpoint で確認したURLを設定
echo "NEXT_PUBLIC_API_URL=https://u2wo63rb4m.execute-api.ap-northeast-1.amazonaws.com" > .env.local
echo "NEXT_PUBLIC_CDN_URL=https://d2b2m401smddic.cloudfront.net" >> .env.local

# 開発サーバー起動
npm run dev
```

ブラウザで `http://localhost:3000` にアクセスする。  
APIリクエストは実際のAWS環境（Lambda + DynamoDB）に対して行われる。

## 7. デプロイチェックリスト

### バックエンド（Lambda）変更時

- [ ] `backend/lambda/handler.rb` の変更を確認
- [ ] `Compress-Archive` で zip 作成
- [ ] `aws lambda update-function-code` で Lambda 更新
- [ ] `/health` エンドポイントで動作確認
- [ ] APIエンドポイントで動作確認（GET /api/v1/recipes など）

### フロントエンド（Next.js）変更時

- [ ] `npm run build` でビルド成功を確認（型エラーがないこと）
- [ ] `aws s3 sync out s3://...` でS3にアップロード
- [ ] `aws cloudfront create-invalidation` でキャッシュ無効化
- [ ] ブラウザで動作確認（新規登録・検索・買い物リストなど）

### インフラ（Terraform）変更時

- [ ] `terraform plan` で変更内容を確認（意図しない削除がないこと）
- [ ] `terraform apply` で適用
- [ ] 出力値（endpoint / bucket名 / distribution ID）を確認

## 8. トラブルシューティング

### CloudFront が古いファイルを返す

キャッシュが残っている可能性がある。

```powershell
aws cloudfront create-invalidation --distribution-id <id> --paths "/*"
```

### Lambda が 500 を返す

CloudWatch Logs で原因を確認する。

```powershell
aws logs tail /aws/lambda/recipe-app-api --since 5m
```

### S3 への画像アップロードが失敗する

1. Lambda の `/api/v1/upload` が Presigned URL を正しく返しているか確認
2. S3 バケットのCORS設定が正しいか確認（`terraform apply` 済みか確認）

### PowerShell で HTTPS リクエストが失敗する

PowerShell 5.1 はデフォルトでTLS 1.0/1.1を使用する場合がある。

```powershell
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
# 上記を同じコマンドチェーンで実行してから Invoke-RestMethod を実行
```
