| 技術 | バージョン | 選定理由 |

| Next.js | 16.2.7 | App Router + `output: "export"` でS3直接ホスティング可能。TypeScriptとの親和性が高い |

| 技術 | バージョン | 選定理由 |
|------|-----------|---------|
| AWS Lambda | - | サーバレス。EC2の起動・管理・パッチ適用が不要。個人利用ではほぼ無料 |
| Ruby | 3.2 | 学習目的でRailsの経験があるため採用。シンプルなAPIならRubyで十分 |

### データストア

| 技術 | 選定理由 |
|------|---------|
| Amazon DynamoDB | サーバレス・マネージド。RDSと異なりVPCやサブネット設定が不要。PAY_PER_REQUEST課金でほぼ無料 |
| Amazon S3（画像） | オブジェクトストレージとして画像を永続化。Presigned URLでブラウザから直接アップロード可能 |

### インフラ・配信

| 技術 | 選定理由 |
|------|---------|
| Amazon CloudFront | CDN + HTTPS終端。フロントエンドS3・API Gateway・画像S3を単一ドメインで統合配信 |
| Amazon S3（フロント） | 静的ファイルのホスティング。EC2不要でコスト最小 |
| Terraform | IaC。環境を再現可能な形でコード管理。`terraform apply` 一発でインフラ構築 |

## 2. アーキテクチャ選定の背景

### EC2/Rails/RDS 構成からの移行

当初はEC2（Rails）+ RDS（MySQL）の構成で設計していた。  
しかし以下の理由からサーバレス構成に移行した。

| 課題（旧構成） | 解決策（新構成） |
|--------------|---------------|
| EC2の最低コスト約$15〜25/月 | Lambda・DynamoDB・API Gatewayは個人利用でほぼ$0/月 |
| EC2のOSパッチ・再起動管理が必要 | Lambda・DynamoDB はAWS管理。運用コストゼロ |
| RDSはVPC・サブネット・SG設定が必要 | DynamoDBはVPC不要。Terraform構成が大幅にシンプル化 |
| EC2が止まるとAPIが全停止 | Lambda は自動スケール・高可用性 |

## 3. 各選定の詳細理由

### Next.js（静的エクスポート）を選んだ理由

- `output: "export"` を設定することでSSR不要のSPAとして出力できる
- S3に `out/` ディレクトリをそのままアップロードすればホスティング完了
- サーバ（Node.jsランタイム）が不要なため、LambdaやEC2なしで動く

### DynamoDB を選んだ理由

- サーバレス構成との相性が良い（Lambda から SDK で直接アクセス）
- VPC・サブネット・セキュリティグループが不要
- PAY_PER_REQUEST課金 → 月数十アクセス程度なら実質$0
- スキーマレスで `ingredients`, `steps` などのネストしたデータを自然に扱える

### Ruby（Lambda）を選んだ理由

- RaiseTech学習課題でRailsの経験を活かすため
- シンプルなCRUD APIであれば標準ライブラリ + AWS SDK for Ruby だけで実装可能
- Gemfileなし・依存ライブラリなしで Lambda zip が最小サイズになる

### CloudFront を選んだ理由

- 単一ドメインでフロントエンド・API・画像の3つのオリジンを統合
- `/*` → S3, `/api/*` → API Gateway, `/images/*` → S3(画像) を振り分け
- API Gateway に直接アクセスするとCORS問題が発生しやすいため、CloudFront でラップ
- HTTPS終端・キャッシュ制御・OACによるS3保護を一括管理

### 旧構成（EC2/Rails/RDS）との比較

| 項目 | 旧構成（EC2+RDS） | 新構成（サーバレス） |
|------|-----------------|------------------|
| 月次コスト | 〜$25/月 | 〜$0/月 |

| 可用性 | EC2単一障害点 | Lambda自動スケール |
